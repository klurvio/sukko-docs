# Testing

Sukko ships with a built-in tester service and CLI commands for validating correctness, measuring performance, and catching regressions — locally or against remote deployments.

## How the Testing Stack Works

The `sukko test` CLI commands are the orchestration layer. They talk to the **tester service** (`sukko-tester`, port 8090), which is the execution layer. The tester service connects to your gateway and provisioning API, runs the selected suite or test type, and streams results back to the CLI.

```
sukko test validate --suite auth
       │
       ▼
 Tester Service (port 8090)
       │
       ├── Gateway (WebSocket connections, auth validation)
       └── Provisioning API (tenant lifecycle, key management)
```

The tester runs as part of the Sukko stack — started automatically by `sukko up` or via Helm. You never need to call it directly; the CLI wraps everything.

### Output signal vocabulary

Every suite produces a two-level result:

| Level | Field | Values | Meaning |
|-------|-------|--------|---------|
| Suite | `Report.Status` | `pass` / `fail` / `error` | Overall outcome. `error` means the suite couldn't run (e.g., admin key missing). `pass` with `[skip]` checks is still a pass. |
| Check | `CheckResult.Status` | `pass` / `fail` / `skip` | Per-check result. `skip` means the check was bypassed — typically because the current edition doesn't include the feature. |

A suite returns `pass` even when edition-gated checks are skipped. Use `fail` or `error` to gate CI; `skip` is expected behavior on lower editions.

## Test Types

| Type | Purpose | Default Duration |
|------|---------|-----------------|
| **Smoke** | Quick connectivity and health check | Instant |
| **Validate** | End-to-end correctness suites | Per suite |
| **Load** | Sustained connections with configurable publish rate | 5 minutes |
| **Stress** | Push to maximum capacity and find limits | 10 minutes |
| **Soak** | Long-running stability — memory leaks, connection drift | 2 hours |

## Contexts: Local vs Remote

Tests run against whichever environment your active context points to. Switch environments with a single command — no config files to edit.

### Setting Up Contexts

After `sukko init`, a `local` context is created automatically:

```bash
sukko context current
```

```
Name:            local
Gateway:         ws://localhost:3000
Provisioning:    http://localhost:8080
Tester:          http://localhost:8090
Environment:     local
Tenant:          local
```

To test against a remote deployment, create a context:

```bash
sukko context create staging \
  --gateway-url wss://ws.staging.example.com \
  --provisioning-url https://api.staging.example.com \
  --tester-url https://tester.staging.example.com \
  --environment staging \
  --tenant my-tenant
```

### Switching Contexts

```bash
sukko context use staging
sukko --context staging test smoke
sukko context list
```

### Local vs Remote Behavior

- **Local context** (localhost URLs) — the tester uses its own Docker-internal service addresses. No context is sent to the tester API.
- **Remote context** — the CLI forwards gateway URL, provisioning URL, and environment to the tester so it targets the right deployment.

## Before Running Suites

### Setup Order Matters

Run these steps in order the first time you bring up a local deployment:

```bash
sukko auth keygen        # Generate admin keypair — MUST run before sukko up
sukko up                 # Start the stack
sukko health             # Wait until all services report healthy
sukko status             # Confirm all services are running
sukko edition            # Check your edition and license key path
sukko test smoke         # Final gate — run this before any suite battery
```

:::warning Admin keypair must exist before the stack starts
If you run `sukko up` before `sukko auth keygen`, the provisioning service starts without a bootstrap admin key. The `provisioning`, `tenant-isolation`, and `token-revocation` suites will fail at setup with no obvious error. If this happens, run `sukko auth keygen`, then `sukko down && sukko up` to restart with the key in place.
:::

### Admin Keypair Requirement

Three suites require an admin keypair to sign provisioning API requests:

| Suite | Requires Admin Key |
|-------|--------------------|
| `provisioning` | yes |
| `tenant-isolation` | yes |
| `token-revocation` | yes |
| All other suites | no |

The tester service looks for the key at `TESTER_ADMIN_KEY_FILE`. In local Docker, `sukko auth keygen` writes the key to `~/.sukko/admin.key` and `sukko up` mounts it automatically — no manual config needed.

For remote deployments, set `TESTER_ADMIN_KEY_FILE` on the tester service to the path of the key file on that host.

### TESTER_ADMIN_KEY_FILE Format

The file must contain the **raw 64-byte Ed25519 binary private key** — not a PEM-encoded PKCS#8 key. The common failure mode is exporting a key from OpenSSL as PKCS#8 PEM, which causes a parse error with no helpful message.

The correct format (produced by `sukko auth keygen`) is:
```
64 raw bytes: 32-byte seed || 32-byte public key
```

To verify:
```bash
wc -c ~/.sukko/admin.key   # must print exactly 64
```

### Edition Requirements

Two suites require a specific edition:

| Suite | Minimum Edition |
|-------|----------------|
| `token-revocation` | Pro <EditionBadge edition="pro" /> |
| `push` | Enterprise <EditionBadge edition="enterprise" /> |

Running a gated suite on the wrong edition returns a `pass` result with one `[skip]` check (the edition gate). No failures — the suite just skips the gated checks. To actually validate the feature, you need a valid license key for the required edition.

## Validation Suites

Run any suite with:

```bash
sukko test validate --suite <name> --follow
```

### Suite run order

Run all 18 suites in this order. Each suite is independent and can be run individually.

| # | Suite | Min Edition | Admin Key | Description |
|---|-------|-------------|-----------|-------------|
| 1 | `smoke` | Community | no | Basic connectivity gate |
| 2 | `auth` | Community | no | JWT auth validation |
| 3 | `channels` | Community | no | Channel subscribe/unsubscribe lifecycle |
| 4 | `pubsub` | Community | no | Pub-sub delivery with channel scoping |
| 5 | `ordering` | Community | no | FIFO message ordering |
| 6 | `reconnect` | Community | no | Disconnect/reconnect recovery |
| 7 | `ratelimit` | Community | no | Rate limit enforcement |
| 8 | `sse` | Community | no | SSE transport validation |
| 9 | `rest-publish` | Community | no | REST publish delivery |
| 10 | `provisioning` | Community | **yes** | Provisioning API CRUD lifecycle |
| 11 | `tenant-isolation` | Community | **yes** | Cross-tenant message isolation |
| 12 | `token-revocation` | **Pro** | **yes** | Token revocation force-disconnect |
| 13 | `edition-limits` | Community | no | Edition boundary limits |
| 14 | `push` | **Enterprise** | no | Push notification pipeline |
| 15 | `license-reload` | Community | no | License hot-reload propagation |
| 16 | `api-key` | Community | no | Static API key auth in isolation |
| 17 | `upgrade` | Community | no | Auth upgrade flow: API key → JWT |
| 18 | `webhooks` | **Pro** | no | Webhook delivery: happy-path, retry, degraded |

### Operator-Relevant Suites

These suites validate features you configure and operate. Run them as part of your deployment checklist.

#### `provisioning`

```bash
sukko test validate --suite provisioning --follow
```

Validates the provisioning API end-to-end: create and delete tenants, register and revoke signing keys, create API keys, create and delete routing rules, and verify that provisioned tenants can connect and receive messages.

**Requires admin keypair** — the suite calls admin-only endpoints.  
**Minimum edition**: Community

#### `pubsub`

```bash
sukko test validate --suite pubsub --follow
```

Validates message delivery with channel scoping. Publishes to public, user-scoped (`dm.{sub}`), and group-scoped (`room.{group}`) channels. Verifies that each subscriber receives exactly the messages they should and no messages they shouldn't.

**Minimum edition**: Community

#### `token-revocation` <EditionBadge edition="pro" />

```bash
sukko test validate --suite token-revocation --follow
```

Validates the token revocation feature: revoke a JWT by token ID (`jti`) and by subject (`sub`), then verify that the revoked token is rejected on a new connection. Also verifies that valid tokens are unaffected.

On Community edition, the gated checks are skipped and the suite reports `pass` with one `[skip]`. On Pro+, all checks run.

**Requires admin keypair** (to provision the test tenant).  
**Minimum edition**: Pro for full validation

#### `sse`

```bash
sukko test validate --suite sse --follow
```

Validates the SSE transport: connects as an SSE client, subscribes to channels, verifies message delivery. Confirms SSE clients and WebSocket clients on the same channel both receive messages.

**Minimum edition**: Community

#### `rest-publish`

```bash
sukko test validate --suite rest-publish --follow
```

Validates the REST publish endpoint: publishes a message via HTTP and verifies it is delivered to WebSocket subscribers on the target channel.

**Minimum edition**: Community

#### `edition-limits`

```bash
sukko test validate --suite edition-limits --follow
```

Validates that edition limits are enforced: verifies tenant count limits, connection count limits, and shard limits match the active edition. Does not test `MaxTopicsPerTenant`.

**Minimum edition**: Community

#### `push` <EditionBadge edition="enterprise" />

```bash
sukko test validate --suite push --follow
```

Validates the push notification pipeline: device registration, subscription CRUD, credential management, and — when a mock receiver is configured — **actual Web Push delivery**. The delivery check registers a subscription against the tester's built-in mock WebPush receiver with real P-256 client keys, publishes a message, and asserts the encrypted notification arrives (RFC 8030). The delivery check runs only when `TESTER_PUSH_RECEIVER_HOST` points at the receiver (it skips on managed deployments where no receiver is reachable); the rest of the suite runs regardless. This delivery path is exercised continuously by the `e2e-push-validate` CI job against a kafka-mode, Enterprise-licensed stack.

:::info Real delivery is Web Push only
End-to-end delivery is verified for **Web Push** (VAPID) only. The FCM (Android) and APNs (iOS) legs register a device with a fake provider token and assert the subscription is accepted — they do not deliver to Google/Apple (that requires live external services). A green suite means Web Push is delivered and verified; mobile push is registered but not delivered.
:::

On Community or Pro, the suite is skipped.  
**Minimum edition**: Enterprise

#### `license-reload`

```bash
sukko test validate --suite license-reload --follow
```

Validates hot license reload: loads a new license key via the webhook endpoint, verifies the edition changes, then restores the original license. Requires `TESTER_SIGNING_KEY_FILE` — a separate Ed25519 private key for signing test license tokens.

**Minimum edition**: Community (tests the reload mechanism, not a specific edition)

#### `api-key`

```bash
sukko test validate --suite api-key --follow
```

Validates static API key authentication in isolation — no JWT provisioning. Confirms that a client connecting with a valid API key is accepted, and that an invalid or missing key is rejected.

**Prerequisites**: Set `TESTER_AUTH_MODE=api-key` and supply a pre-provisioned key via `TESTER_API_KEY` before running this suite. The key must belong to the tenant the tester is configured for.

**Minimum edition**: Community

#### `upgrade`

```bash
sukko test validate --suite upgrade --follow
```

Validates the auth upgrade flow: the client connects with a static API key, sends an `auth` message containing a JWT, and asserts that the server responds with `auth_ack` and grants access to private channels that require a JWT. This suite is the canonical test for the dual-credential connection pattern.

**Prerequisites**: Set `TESTER_AUTH_MODE=upgrade` and supply `TESTER_API_KEY`. The upgrade timeout is controlled by `TESTER_AUTH_UPGRADE_TIMEOUT` (default 10s, max 60s).

**Minimum edition**: Community

#### `webhooks`

```bash
sukko test validate --suite webhooks --follow
```

Validates the full webhook delivery pipeline across three scenarios:

| Scenario | Behaviour | Expected |
|----------|-----------|----------|
| Happy-path | Endpoint always returns 200 | 1 delivery, HMAC valid |
| Retry-success | First 2 attempts → 500, third → 200 | 3 deliveries total |
| Degraded | Endpoint always returns 500 (2 max retries) | Webhook status transitions to `degraded` |

Each scenario provisions a throw-away routing rule and webhook, publishes a message via REST, and polls the tester's built-in `/webhook-receive/{runID}` endpoint to assert delivery count, HMAC signature, and response status codes.

**Prerequisites**: Set `TESTER_WEBHOOK_BASE_URL` to the externally-reachable URL of this tester pod (e.g., `http://tester.sukko-ci.svc.cluster.local:8085`). The webhook-worker must be able to reach this URL to deliver payloads.

**Minimum edition**: Pro. Requires Kafka backend (`MESSAGE_BACKEND=kafka`).

**Skip behavior**: When `TESTER_WEBHOOK_BASE_URL` is not set, or the current edition is Community, all checks return `[skip]` and the suite still reports `pass`.

#### `kafka-ingest`

```bash
sukko test validate --suite kafka-ingest --follow
```

Publishes a message **directly to Kafka/Redpanda** — bypassing the gateway and ws-server's producer — and verifies a gateway-subscribed client receives it. This exercises the ingestion pipeline (Kafka → ws-server consumer → broadcast → client) in isolation, matching how production data actually arrives (from an upstream producer, not a WebSocket client). It is the only tester suite that connects to the message backend itself; every other suite publishes through the gateway.

To reach a managed Kafka (Confluent Cloud, Redpanda Cloud, AWS MSK, Aiven), configure the tester's Kafka connection env vars: `KAFKA_BROKERS`, `KAFKA_SASL_ENABLED`, `KAFKA_SASL_MECHANISM` (`plain` for Confluent Cloud, `scram-sha-256` for Redpanda Cloud/Aiven, or `scram-sha-512` for AWS MSK), `KAFKA_SASL_USERNAME`, `KAFKA_SASL_PASSWORD`, `KAFKA_TLS_ENABLED`, and optionally `KAFKA_TLS_CA_PATH`. To override the broker addresses for a single run, pass `--kafka-brokers`. Credentials always come from the tester's environment — never from the CLI.

**Prerequisites**:
- The **server under test must run `MESSAGE_BACKEND=kafka`** — otherwise records dropped into Kafka are never consumed and the suite times out (the failure message includes this hint).
- The tester and the server under test must resolve to the **same Kafka topic namespace** — set the same `ENVIRONMENT` (and the same `KAFKA_TOPIC_NAMESPACE_OVERRIDE`, if used) on both.

**Minimum edition**: Community.

**Skip behavior**: When `KAFKA_BROKERS` is not set, the suite returns `[skip]` and the run still reports `pass`.

### Developer-Internal Suites

These suites validate core WebSocket infrastructure. They run without an admin keypair and pass on Community. Include them in automated CI but they rarely need manual attention:

| Suite | What it validates |
|-------|------------------|
| `auth` | JWT validation — expired tokens, wrong key, wrong tenant, revoked keys |
| `channels` | Channel subscribe and unsubscribe operations |
| `ordering` | Message FIFO ordering — messages arrive in publish order |
| `reconnect` | Disconnect/reconnect recovery with message replay |
| `ratelimit` | Rate limit enforcement — clients throttled at configured limits |
| `tenant-isolation` | Cross-tenant isolation — tenant A cannot see tenant B's messages (requires admin key) |

## Load, Stress, and Soak Tests

### Load Test

Sustained load with configurable connections and publish rate. Use this to verify your deployment handles expected traffic.

```bash
sukko test load \
  --connections 100 \
  --duration 5m \
  --publish-rate 10 \
  --ramp-rate 5 \
  --follow
```

### Stress Test

Push beyond expected capacity to find breaking points and verify graceful degradation.

```bash
sukko test stress \
  --connections 1000 \
  --duration 10m \
  --ramp-rate 100 \
  --follow
```

### Soak Test

Long-running stability test. Catches memory leaks, connection drift, and JWT refresh failures.

```bash
sukko test soak \
  --connections 100 \
  --duration 2h \
  --publish-rate 10 \
  --follow
```

| Flag | Load/Stress/Soak | Description |
|------|-----------------|-------------|
| `--connections` | all | Number of WebSocket connections |
| `--duration` | all | Test duration (`30s`, `5m`, `2h`) |
| `--publish-rate` | load, soak | Messages per second |
| `--ramp-rate` | load, stress | Connections per second during ramp-up |

### Streaming Output

Add `--follow` (or `-f`) to stream metrics in real time:

```bash
sukko test load --connections 100 --duration 5m --follow
```

For machine-parseable output:

```bash
sukko test load --connections 100 --duration 5m --follow --output json
```

## Deployment Validation Checklist

Run these steps in order after deploying to a new environment or after a significant configuration change.

**Before starting**: Confirm you have run `sukko auth keygen` before the stack was first started. If not, see the admin keypair setup below.

### Step 1 — Environment cleanup

If re-validating an existing environment, remove leftover tester tenants from a previous run:

```bash
# List all tenants
sukko tenants list

# Delete any test tenants (prefixed with tester- or test-)
sukko tenants deprovision tester-pubsub --force
sukko tenants deprovision tester-isolation-a --force
# ... repeat for each leftover test tenant
```

### Step 2 — Health check

```bash
sukko health
sukko status
```

All services must report healthy before proceeding. If any service is unhealthy, investigate logs before running tests.

### Step 3 — Smoke test

```bash
sukko test smoke --follow
```

The smoke test must pass before running any other suite. If it fails, fix the underlying connectivity issue first.

### Step 4 — Admin keypair setup (local)

For local Docker deployments managed by `sukko up`, the admin key is mounted automatically from `~/.sukko/admin.key`. Verify it is present:

```bash
wc -c ~/.sukko/admin.key   # must print 64
```

If missing, run `sukko auth keygen` then restart: `sukko down && sukko up`.

### Step 5 — Admin keypair setup (remote)

For remote/staging deployments, the admin key must be registered in provisioning and the tester must be configured to use it:

```bash
# Register the admin public key (run once per environment)
sukko auth register --env staging

# Verify the tester service has TESTER_ADMIN_KEY_FILE set to the
# corresponding private key file on its host
```

### Step 6 — Core suite battery

Run the operator-relevant suites in order:

```bash
sukko test validate --suite provisioning --follow
sukko test validate --suite pubsub --follow
sukko test validate --suite tenant-isolation --follow
sukko test validate --suite sse --follow
sukko test validate --suite rest-publish --follow
sukko test validate --suite edition-limits --follow
```

### Step 7 — Edition-gated suites

Run only the suites your edition supports:

```bash
# Pro+ only
sukko test validate --suite token-revocation --follow

# Enterprise only
sukko test validate --suite push --follow
```

### Step 8 — Auth mode suites (optional)

If your deployment uses API key auth or the auth upgrade flow, run:

```bash
# Requires TESTER_AUTH_MODE=api-key and TESTER_API_KEY
sukko test validate --suite api-key --follow

# Requires TESTER_AUTH_MODE=upgrade and TESTER_API_KEY
sukko test validate --suite upgrade --follow
```

### Step 9 — Webhooks suite (Pro+)

```bash
# Requires TESTER_WEBHOOK_BASE_URL and MESSAGE_BACKEND=kafka
sukko test validate --suite webhooks --follow
```

All checks are automatically skipped (suite still passes) when the base URL is not set or the edition is Community.

**Excluded from standard checklist** (developer-internal, automated CI only): `auth`, `channels`, `ordering`, `reconnect`, `ratelimit`, `license-reload`.

## Troubleshooting

### Tester service unreachable

**Symptom**: `sukko test smoke` returns `connect: connection refused` or times out immediately.

**Fix**: The tester service is not running or not reachable at the configured URL.
- Local: `sukko status` — check that the tester service shows healthy. If not, `sukko down && sukko up`.
- Remote: verify `TESTER_URL` in the CLI context points to the correct host and port. The tester listens on port 8090 by default.

### Suite fails with 403 EDITION_LIMIT

**Symptom**: A suite reports a check failure with message `This feature requires pro edition or higher` (or enterprise).

**Fix**: You are running a gated suite on an edition that does not support it. Either:
- Load a valid Pro/Enterprise license: `sukko license push --file <license.key>`
- Accept that the suite will skip the gated checks and report `pass` with `[skip]`

### Routing rules rejected with 400 TOPIC_NOT_PROVISIONED

**Symptom**: The `provisioning` suite fails when creating a routing rule. Error: `TOPIC_NOT_PROVISIONED`.

**Fix**: In the current release, routing rules can only reference topics that already exist (`default` and `dead-letter` are always available). A routing rule referencing any other topic suffix will be rejected until that topic is explicitly provisioned. The suite should only reference `default` — if it references other topics, the tester version may not match the provisioning API version.

### pubsub or tenant-isolation fails with 400 (version mismatch)

**Symptom**: Suite fails immediately with HTTP 400 and no descriptive message.

**Fix**: The tester service version does not match the provisioning API version. Ensure the tester and provisioning images are from the same release. Check image tags: `sukko status --verbose`.

### license-reload suite fails at signing step

**Symptom**: `license-reload` suite fails with a key parsing or signature error.

**Fix**: `TESTER_SIGNING_KEY_FILE` on the tester service is not set, is set to the wrong file, or contains a key in the wrong format (must be raw 64-byte Ed25519, same format as `TESTER_ADMIN_KEY_FILE`). Verify the key is present and correct format (`wc -c` prints 64).

### license-reload suite fails to restore original license

**Symptom**: The `license-reload` suite passes the reload check but fails the restore step. After the suite, your deployment may be running on a test edition.

**Fix**: The suite loads a test license then tries to reload the original. If the original is no longer accessible (e.g., `SUKKO_LICENSE_KEY` is not set), the restore fails. Set `SUKKO_LICENSE_KEY` in the tester service environment so it can restore. If the deployment is stuck on the wrong edition after a failed restore, reload the correct license manually: `sukko license push --file <license.key>`.

### RequireAdminProvider: admin keypair not found

**Symptom**: The `provisioning`, `tenant-isolation`, or `token-revocation` suite fails at the very first check with a message about missing admin credentials.

**Fix**: `TESTER_ADMIN_KEY_FILE` is not set on the tester service, or the file does not exist at the specified path.
- Local: the key should be mounted automatically by `sukko up`. If missing, run `sukko auth keygen` then restart.
- Remote: set `TESTER_ADMIN_KEY_FILE` to the path of the raw 64-byte admin private key on the tester host.

### edition-limits suite leaves orphaned test tenants

**Symptom**: After the `edition-limits` suite, `sukko tenants list` shows tenants prefixed with `tester-limits-`.

**Fix**: The suite creates test tenants and normally cleans them up. If it fails mid-run, tenants are left behind. Clean up manually:

```bash
sukko tenants deprovision tester-limits-1 --force
sukko tenants deprovision tester-limits-2 --force
# ... repeat as needed
```

### Smoke test times out

**Symptom**: `sukko test smoke` hangs for 30 seconds then returns a timeout error.

**Fix**: The gateway is not accepting WebSocket connections. Check:
1. `sukko health` — all services healthy?
2. Is the gateway port (default 3000) accessible from the tester service?
3. Are there firewall rules blocking WebSocket upgrade requests?

### Context points to wrong environment

**Symptom**: Tests pass locally but fail against the remote context, or vice versa — in ways that suggest the wrong URL is being used.

**Fix**: Confirm the active context:
```bash
sukko context current
```
Switch if needed: `sukko context use <name>`. For a one-off test against a different environment without switching: `sukko --context staging test smoke`.

### Admin key silently ignored (catch-22)

**Symptom**: The stack is running but the `provisioning` suite fails at admin setup even though `TESTER_ADMIN_KEY_FILE` is set correctly.

**Root cause**: `sukko up` was run before `sukko auth keygen`. The provisioning service started without `ADMIN_BOOTSTRAP_KEY` and registered no admin key. The tester has a valid private key but it has no corresponding public key registered in provisioning, so admin JWTs are rejected.

**Fix**: Run `sukko auth keygen` to generate the keypair, then restart the stack so provisioning starts with the bootstrap key: `sukko down && sukko up`.

### TESTER_ADMIN_KEY_ID mismatch

**Symptom**: The `provisioning`, `tenant-isolation`, or `token-revocation` suite fails at admin setup with a JWT validation error (`wrong_kid` or `invalid_key_id`), even though `TESTER_ADMIN_KEY_FILE` is set and the file contains 64 bytes.

**Root cause**: The `kid` (key ID) in JWTs issued by the tester must match the ID under which the public key was registered in the provisioning service. `TESTER_ADMIN_KEY_ID` defaults to `bootstrap-0`. If the key was registered under a different ID, JWTs are rejected with `wrong_kid`.

**Fix**: Set `TESTER_ADMIN_KEY_ID` to the key ID used during `sukko auth keygen` (shown in the output). The default is `bootstrap-0` when `sukko auth keygen` is run without flags.

### TESTER_ADMIN_KEY_FILE missing at startup

**Symptom**: The tester service fails to start entirely — it does not serve any HTTP requests and the container exits immediately with a startup error referencing the admin key file.

**Root cause**: `TESTER_ADMIN_KEY_FILE` is set to a path that does not exist on the tester container. The tester validates the key file path at startup and exits if the file cannot be opened. This is a startup failure, not a test failure — suites will not run at all.

**Fix**: Verify the file exists at the configured path on the tester container or pod. In local Docker, `sukko up` mounts `~/.sukko/admin.key` automatically — run `sukko auth keygen` first, then `sukko down && sukko up`. On Kubernetes, confirm the secret volume or ConfigMap is mounted at the correct path.

### Webhooks suite skips everything

**Symptom**: `sukko test validate --suite webhooks` reports `pass` with all checks returning `[skip]`.

**Cause**: Either (a) `TESTER_WEBHOOK_BASE_URL` is not set, or (b) the current edition is Community. Both conditions cause the suite to skip all checks rather than fail.

**Fix**: For (a), set `TESTER_WEBHOOK_BASE_URL` to the externally-reachable HTTP URL of the tester pod — the webhook-worker must be able to reach this URL to deliver payloads. For (b), the webhooks suite requires Pro edition or higher.

### Orphaned tester tenants from previous runs

**Symptom**: The `pubsub`, `provisioning`, or `tenant-isolation` suite fails immediately with `409 TENANT_ALREADY_EXISTS`.

**Fix**: A previous test run was interrupted and left test tenants behind. Clean up:
```bash
sukko tenants list
# Delete any tenants prefixed with tester-
sukko tenants deprovision <tester-tenant-id> --force
```
